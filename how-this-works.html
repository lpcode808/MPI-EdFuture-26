How this guide works

A behind-the-scenes tour of the EdFuture 2026 pocket program — how a small pile of plain files becomes an app that renders a schedule, remembers your notes, and keeps working offline. Written for the technically curious; every term of art gets a plain-language gloss.

01 What is this site?

It's an unofficial pocket program for the 2026 EdFuture Summit at Mid-Pacific Institute. One page, four tabs: browse the two-day Program, look up who's presenting in People, keep private Notes, and check Info. Because the summit runs single-track — everyone's in the same room — there's nothing to plan around, so the guide skips bookmarking and puts the weight on search, the people directory, and notes.

The defining constraint: everything you write stays on your device. There is no account, no login, no analytics, and no server that ever sees your notes. That isn't a limitation that snuck in — it's the design.

Mental model Think of it as a paper program with a pencil attached. The printed part (the schedule) is the same for everyone and comes from the event's published flyers. The pencil marks (your notes) exist only on your copy. Nobody collects the pencils.

02 The big picture: static files, zero dependencies

The whole app is plain HTML, CSS, and JavaScript — no framework, no build tool that transforms code, no node_modules folder. "Zero runtime dependencies" means the only software the app relies on is the browser you're already holding.

A typical conference app

Native app store download, or a React site talking to a backend. Accounts, push notifications, analytics.

Needs servers, maintenance, and your data.

This guide

Sixteen files served from GitHub Pages. The browser downloads them once and can run them offline.

Needs nothing, knows nothing about you.

It is, however, a PWA — a progressive web app. That's a website with two extra powers: it can be installed to your home screen like a native app, and it can keep working without a network connection. Both powers come from small standard files, not from an app store.

03 The files

Here's the project, starred where it matters. Note the split: the top group ships to attendees; the bottom group is workshop material that never leaves the repository.

MPI-EdFuture-26/ ├── index.html ← the shell: header, four tab panels, footer ├── app.js ← the brain: tabs, search, filters, people, notes ├── storage.js ← how your notes are saved on-device ├── data/schedule.js ← all 29 program items, transcribed from flyers ├── data/event.js event facts (dates, venue, official URL) ├── styles.css the evergreen night-mode visual system ├── sw.js service worker — the offline machinery ├── manifest.webmanifest "install me" metadata for phones ├── owl.js 🦉 (see section 09) ├── how-this-works.html you are here ├── assets/ banner, icons, QR code, one pixel font │ │ — never deployed — ├── scraped/ ← source evidence: the two program flyers ├── scripts/ validators + the allowlisted build ├── docs/ architecture, privacy, roadmap notes └── AGENTS.md the contract AI agents read before editing

Three things worth noticing:

  1. Data is code here. The schedule isn't fetched from an API — it's a JavaScript file (data/schedule.js) exporting a frozen array of 29 session objects. Editing the program means editing that file and republishing.
  2. Every session carries provenance. Each item records which source flyer it came from. A validation script fails if a session claims information the sources never published.
  3. The scraped/ folder is evidence, not content. It holds the two public program images the schedule was transcribed from, so anyone can audit the transcription — but it's excluded from what gets deployed.

04 How the program gets on screen

When the page loads, this happens:

01Browser loads index.html
02app.js imports the schedule
03A template is cloned per session
04Cards are grouped by day
05You search, filter, note

index.html contains a <template> element — an inert chunk of HTML the browser parses but doesn't display. It's the blank session card: a time slot, a title, a room, a hidden note field. For each of the 29 sessions, app.js clones the template, fills in the fields, and appends it to the list — the Program tab clones it once per session, and the Notes tab reuses the exact same function to render whichever sessions already have a note attached:

const node = template.content.firstElementChild.cloneNode(true);
node.querySelector(".session-time").textContent = session.time;
node.querySelector("h3").textContent = session.title;
// … room, presenters, note state …

Search and day filters don't talk to a server either. Typing in the search box just re-runs a filter over the in-memory array — checking your query against each session's title, room, type, and people — and re-renders the list. With 29 items, that's instant. The People tab builds its directory the same way: it walks every session's presenter list once, groups repeat names, and tapping a name reuses that same search filter to jump back to their session.

The four tabs are one page wearing four outfits. Clicking a tab hides three panels and shows one, and writes the choice into the URL after the # symbol (#agenda, #people, #notes, #info) so a shared or reloaded link lands on the right tab. The tabs are real role="tab" controls with full arrow-key navigation — accessibility here is load-bearing, not decorative: 44px touch targets, visible focus rings, reduced-motion support, and status text that screen readers announce.

05 Where your notes live

Write a note on any session and it's saved via localStorage — a built-in browser feature that lets a site keep small amounts of text on your device. The entire personal state is one JSON blob under one key:

// key: mpi_edfuture_26_attendee_state_v2
{
  "schemaVersion": 2,
  "sessionNotes": { "d2-keynote": "ask about rubric design" },
  "quickNotes": "find the TechZone table at lunch"
}

storage.js wraps this with two disciplines. First, a schema version baked into the key, so a future redesign of the format can't misread old data — and if it finds nothing under the current key, it looks for the previous version's key once and carries your notes forward instead of quietly losing them. Second, a normalize step: anything read from storage — or imported from a backup file — is filtered down to exactly the expected shape. Unknown fields are dropped, wrong types discarded. The app never trusts stored data just because it stored it.

That's also how Export / Import backup works. Export wraps your state in an envelope stamped with a format name and the event ID, and downloads it as a JSON file. Import checks both stamps before accepting anything — a backup from a different event, or a random JSON file, is politely refused.

Honest trade-off Device-local means device-local: your phone and your laptop each have their own notes, and clearing browser data erases them. The backup file is the escape hatch — export on one device, import on the other. That's the price of a guide that collects nothing.

06 Working offline: the service worker

Summit wifi is a shared resource under load. The guide plans for that with a service worker (sw.js) — a small script the browser installs alongside the page that can intercept network requests, like a desk clerk who checks the filing cabinet before calling the archive.

On first visit, the worker stores the whole app shell — HTML, CSS, JavaScript, schedule data, fonts, images — in a named cache. After that it follows a strategy called stale-while-revalidate:

01You request a file
02Cache answers instantly
03A fresh copy is fetched in the background
04Cache is updated for next time

So the app always opens fast (or with no connection at all), and program corrections still reach you — one visit late, which is fine for a schedule. When the shell's file list changes, the cache name is bumped (…-v6, …-v7) and the worker throws away the old cache wholesale rather than reasoning about individual files.

The manifest.webmanifest file completes the PWA story: it gives the browser a name, icons, colors, and a start URL, which is what makes "Add to Home Screen" produce something that opens full-screen like a native app.

07 The source boundary

The most unusual engineering in this project isn't in the JavaScript — it's in what the app refuses to know. The entire content comes from exactly three public sources: the two program flyers published for the event, and the public Humanitix listing.

  • Names appear only when printed on those flyers. No bios were imported, even where public profiles exist.
  • The flyers publish start times but not end times — so the app shows only start times rather than inventing plausible durations.
  • Where the sources contradict each other (the listing disagrees with itself about the event's end time), the conflict is recorded in the source snapshot, not silently resolved.

And this is enforced by machine, not just by intention. scripts/validate-data.mjs checks every session against rules like "every item must cite its source image" and "no session may carry a field the sources can't support." It runs as part of the test suite, so a well-meaning future edit that invents an end time would fail before it shipped.

Why it matters Most software bugs are code that does the wrong thing. In a guide like this, the worse failure is data that says the wrong thing — an invented room, a misattributed name. Treating "don't invent" as a testable rule, with the source images kept in the repo as evidence, makes the honesty auditable.

08 Tests, the build, and going live

Two commands guard the project. npm test is strictly read-only — it changes nothing, it only checks: JavaScript syntax, the data provenance rules above, a storage round-trip (export → import → identical state), and a static smoke test that greps for accessibility-critical markup, the service worker's safety checks, and required PWA assets.

npm run build constructs the deployable site in dist/ from a fixed allowlist — an explicit list of the sixteen attendee-facing files that are permitted to ship. Everything not on the list (source flyers, internal docs, validation scripts, agent notes) is excluded by default. Blocklists forget; allowlists can't leak what they never include.

01Edit files
02npm test
03npm run build
04git push
05GitHub Pages publishes dist/

Hosting is GitHub Pages — free static hosting driven by a GitHub Actions workflow that runs on every push to main, rebuilds dist/ fresh from the allowlist, and publishes only that folder. The workflow's third-party actions are pinned to exact commit hashes (not floating version tags), so a compromised upstream action can't silently swap in new code. Small site, real supply-chain hygiene.

09 The owl

Tap the three-dot brand mark five times fast — or type owl on a keyboard — and the guide opens Night Flight, a read-only pixel-art overworld (a nod to the Owlgorithms Club on the Day 2 program). Your notes appear as glowing landmarks on a night map, drawn as inline SVG sprites with a self-hosted pixel font.

Technically it demonstrates a nice pattern: lazy loading. The easter egg lives in its own module, owl.js, which is only downloaded when summoned:

// the base guide pays no cost for the surprise
function summonOwl() {
  import("./owl.js").then((module) => module.openOwlMode(store));
}

It reads the same program data and your same notes, but never writes anything — the easter egg respects the storage schema by staying out of it.

10 What's next: the roadmap

The project keeps a public roadmap in the repository (docs/roadmap.md), ordered by when the work matters:

  • Before the summit — a real-device shakedown of the live deploy (home-screen install, airplane-mode reload, QR scan, a note/export round trip), and re-transcribing if the organizers publish an updated flyer. The service worker delivers corrections to installed attendees one visit after each push.
  • During the summit — a possible "happening now / up next" strip on the Program tab, built honestly from published start times only ("started at 9:05am" rather than invented durations). Room changes are just a data edit, a test run, and a push.
  • After the summit — archive mode: a quiet "this event has ended" banner, a freeze on data edits, and the guide kept up as a reference. Notes stay readable.
  • Parked — session descriptions, presenter profiles, and calendar export all wait on the organizers publishing the underlying data. The source boundary from section 07 gates the roadmap too: nothing gets built from data the sources don't provide.

The stated bar for adding anything: keeps the zero-dependency runtime, respects the source boundary, and doesn't crowd the pocket-program core.

11 Glossary

  • Static site — a website that is only files (HTML/CSS/JS), with no server-side logic or database.
  • PWA — progressive web app: a website that can be installed to a home screen and work offline.
  • Service worker — a background script that intercepts a site's network requests, enabling caching and offline use.
  • Stale-while-revalidate — answer from cache immediately, refresh the cache in the background for next time.
  • localStorage — a browser feature for storing small amounts of text on the user's device, per site.
  • Normalization — forcing loaded data into an exact expected shape, discarding anything unexpected.
  • <template> — an HTML element holding markup the browser parses but doesn't render, used as a cloneable blank form.
  • ES module — a JavaScript file that declares what it imports and exports; the browser loads the graph natively, no bundler needed.
  • Provenance — a recorded answer to "where did this data come from?", attached to each program item here.
  • Allowlist — an explicit list of what is permitted (here: files allowed to deploy); the opposite of a blocklist.
  • GitHub Pages / Actions — GitHub's free static hosting, and the automation that rebuilds the site on every push.

12 Colophon — how it was actually built

This guide was built conversationally, in a single day, by describing intent to AI coding agents — the same working method behind the Vibe Coding session on the Day 1 program. The human contribution was the part that doesn't automate: the source boundary, the privacy stance, what the app should refuse to do, and the taste calls. The agents wrote the code, and the test suite in section 08 exists partly so future agents can't drift from those decisions.

  • Structure — modular, dependency-free pattern carried over from earlier conference guides built the same way
  • Visual system — evergreen night palette derived from the summit's own public banner artwork; Sora for UI, Source Serif 4 for reading
  • Guardrails — an AGENTS.md contract in the repo tells any future AI agent the rules before it edits anything
  • Hosting — GitHub Pages, published from an allowlisted dist/ build

If you're curious what that workflow feels like, the whole repository is readable at github.com/lpcode808/MPI-EdFuture-26 — start with data/schedule.js (it's just the program, written down), then app.js top to bottom, then storage.js. The three files together are shorter than many README files.
